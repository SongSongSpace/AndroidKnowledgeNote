# 代理模式笔记

## 1. 一句话定义
**代理 = 拦截器 + 转发器**：在不修改原类代码的前提下，通过一个“假”对象来控制对“真”对象的访问，并可在调用前后插入额外逻辑。

## 2. 核心价值
- **解耦**：调用方不直接依赖真实实现
- **增强**：无侵入地添加日志、权限、缓存、延迟加载等横切逻辑
- **统一**：重复逻辑（如性能监控）只需在代理中写一次

## 3. 本质对比

| 直接调用 | 代理调用 |
|---------|---------|
| `new Target().method()` | `proxy.method()` → 代理做额外处理 → 调用真实 `target.method()` |
| 增强逻辑散落在各处 | 增强逻辑集中在代理类 |
| 修改增强需改N处 | 修改增强只改代理类 |

## 4. 典型应用场景
- **权限控制**：调用前检查是否有权限
- **延迟加载**：真正使用时才创建重量级对象
- **日志/监控**：统一记录方法耗时或入参出参
- **远程通信**：像调用本地方法一样调用远程服务（如 Retrofit）
- **事务管理**：方法执行前后自动开启/提交事务（如 `@Transactional`）

## 5. 两种实现方式对比

| 类型 | 实现方式 | 要求 | 典型框架 |
|------|---------|------|----------|
| **静态代理** | 手动为每个类编写代理类 | 无 | 极少手写 |
| **JDK 动态代理** | 运行时生成代理（需反射） | 被代理类必须实现接口 | Spring AOP、Retrofit |
| **CGLIB 动态代理** | 运行时生成子类作为代理 | 无接口要求 | Hibernate、无接口的 Spring Bean |

## 6. 最简代码示例（JDK 动态代理）

```java
// 1. 接口
interface Calc {
    int add(int a, int b);
}

// 2. 真实对象
class RealCalc implements Calc {
    public int add(int a, int b) { return a + b; }
}

// 3. 代理处理器
class LogHandler implements InvocationHandler {
    Object target;
    LogHandler(Object target) { this.target = target; }
    
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        System.out.println("开始方法：" + method.getName());
        Object result = method.invoke(target, args);
        System.out.println("结束方法");
        return result;
    }
}

// 4. 使用
RealCalc real = new RealCalc();
Calc proxy = (Calc) Proxy.newProxyInstance(
    Calc.class.getClassLoader(),
    new Class[]{Calc.class},
    new LogHandler(real)
);
proxy.add(3, 5);  // 自动打印日志
```
## 7. 必要性的判断标准
当你需要为多个类的多个方法统一增加相同逻辑（日志、权限、缓存...），但不想修改原有代码时，代理就是必要且优雅的解决方案。

反之，如果项目简单、逻辑单一、没有**横切关注点**，则不需要强行引入代理。
