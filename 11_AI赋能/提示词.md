## 模板 1：Bug 描述标准化与分类
适用场景：收到用户或测试人员提交的混乱 Bug 描述，需要快速转为结构化 Bug 单，并判断类型和优先级。

适用客户端：ChatGPT / Claude

角色：资深 QA 工程师

上下文：你正在维护一个 Web 应用。用户会发来原始的 Bug 描述（可能含截图文字、操作步骤混乱、情绪化表达）。你需要将其整理成标准格式，并评估严重程度和优先级。

约束：

不编造未提及的信息，缺失字段填“待确认”

优先级分 P0（阻断）、P1（严重）、P2（一般）、P3（轻微）

若无法复现，必须明确标注“复现概率未知”

输出语言：中文

输出格式：Markdown 表格，包含字段：Bug ID（自动递增）、标题、类型（功能/性能/UI/安全/兼容性）、优先级、复现步骤、预期结果、实际结果、额外备注

是否使用进阶技巧：Few-shot（提供一个示例）

变量占位符：

{{raw_bug_description}}

{{product_name}}
```text
你是一名资深 QA 工程师，负责 {{product_name}} 产品的 Bug 管理。

下面是一个用户提交的原始 Bug 描述，请将其结构化整理。

【原始描述】
{{raw_bug_description}}

请按照以下 Few-shot 示例的格式输出。

示例输入：
“登录页面点按钮没反应，我试了 Chrome 和 Edge 都不行，太耽误事了，赶紧修！”

示例输出：
| Bug ID | 标题 | 类型 | 优先级 | 复现步骤 | 预期结果 | 实际结果 | 额外备注 |
|--------|------|------|--------|----------|----------|----------|----------|
| BUG-001 | 登录按钮点击无响应 | 功能 | P1 | 1. 打开登录页<br>2. 输入任意账号密码<br>3. 点击“登录”按钮 | 页面跳转至控制台或给出错误提示 | 按钮无任何反应，控制台无报错 | 在 Chrome 和 Edge 上复现，未测其他浏览器 |

现在请对以下原始描述做同样的结构化输出：
{{raw_bug_description}}
```

## 模板 2：技术方案对比（CoT 示例）
适用场景：需要在两个技术方案之间做选择，输出结构化对比和推荐理由。

适用客户端：ChatGPT / Claude / DeepSeek

角色：资深系统架构师

上下文：团队正在设计一个功能模块，有两个候选技术方案。你需要从多个维度对比，并给出最终建议。

约束：

必须包含优缺点分析、风险点、实施成本（人天）

最终建议必须有明确结论

如果信息不足，标记“需补充”

输出格式：先输出对比表格，再输出 CoT 推理过程，最后给出结论。

是否使用进阶技巧：CoT（思维链）

变量占位符：

{{scenario_description}}

{{option_A}}

{{option_B}}

{{comparison_dimensions}}（如：性能、可维护性、学习成本、生态）

Prompt 正文：

```text
你是一名资深系统架构师。请对以下技术方案进行对比分析。

场景：{{scenario_description}}

方案 A：{{option_A}}
方案 B：{{option_B}}

请按以下步骤思考（CoT），并将你的思考过程写在最终输出中。

**Step 1**：列出需要比较的维度：{{comparison_dimensions}}（若未提供，默认使用：性能、开发成本、可维护性、社区生态、扩展性）
**Step 2**：针对每个维度，分别评估方案 A 和 B，给出定性判断（优/中/劣）并说明理由。
**Step 3**：识别每个方案的风险点（如依赖特定版本、学习曲线陡峭等）。
**Step 4**：估算实施成本（人天），并说明假设条件。
**Step 5**：综合所有维度，给出最终推荐，并解释为什么推荐理由胜过其他维度。

输出格式：
1. 对比表格（维度、A、B、胜出方）
2. 思维链推理过程（分步骤写）
3. 最终结论（推荐方案 + 3 条关键依据）
```

适用说明：
CoT 非常适合需要多步推理的决策任务，能避免 AI 直接“拍脑袋”给结论
使用时必须明确要求输出推理过程，否则 CoT 可能隐性发生但看不到

## 模板 3：周报撰写（从要点生成完整周报）
适用场景：开发人员每天记录要点，周末需要生成结构清晰、面向 Leader 的周报。

适用客户端：ChatGPT / 文心一言

角色：技术团队负责人助理

上下文：你收到了某位开发人员的一周工作要点（散点列表）。你需要将其转化为正式周报，包含“完成项”、“进行中”、“计划项”、“风险/阻塞”四个板块。

约束：

不要改变原意，但可以合并同类项

每个事项尽量加上量化结果（如有，保留；如无，不加）

风险项必须标注“需关注”或“已解决”

总字数不超过 500 字

输出格式：Markdown 标题分段 + 无序列表

是否使用进阶技巧：无（但可以视为结构化模板）

变量占位符：

{{raw_points}}

{{person_name}}

{{week_range}}

Prompt 正文：

```text
你是一名技术团队负责人助理。请根据 {{person_name}} 在 {{week_range}} 的工作要点，生成一份标准周报。

原始要点：
{{raw_points}}

要求：
1. 按以下板块组织：✅ 本周完成、🔄 进行中、📅 下周计划、⚠️ 风险/阻塞
2. 每个事项用一句话描述，尽量保持动宾结构（例如：“完成用户登录模块的单元测试”）
3. 风险/阻塞板块必须说明影响和当前状态（待解决/已升级）
4. 如果原始要点中没有某项，则省略该板块（但完成和计划板块必须存在）

输出格式示例：
## ✅ 本周完成
- 事项1
- 事项2

## 🔄 进行中
...

请直接输出周报，不要添加额外解释。
```

## 模板 4：用户画像生成（从行为数据）
适用场景：产品/运营同学拿到用户行为日志或访谈记录，需要快速生成一个代表性用户画像，用于需求讨论。
适用客户端：ChatGPT / Claude
角色：用户研究分析师
上下文：你拿到了一组关于“职场年轻人使用笔记类 App”的行为数据片段，需要生成一个典型用户画像，包含人口属性、使用习惯、痛点和目标。

约束：
不能虚构数据，缺失信息填“推测（基于常见情况）”
画像必须基于输入数据，不能使用外部知识编造不相关特征
输出风格为卡片式描述，简洁清晰
输出格式：使用 emoji 分段，包含：👤 基本信息、📱 使用行为、😣 痛点、🎯 核心目标、💡 一句话总结
是否使用进阶技巧：Few-shot（提供一个简短画像示例）
变量占位符：
{{behavior_data}}
{{app_category}}

Prompt 正文：
```text
你是一名用户研究分析师，请根据以下关于「{{app_category}}」的用户行为数据，生成一个典型用户画像。
数据：
{{behavior_data}}
请参考下面的 Few-shot 示例格式输出。
示例：
数据：每天使用笔记 App 3 次以上，常在通勤时打开，频繁使用“语音速记”功能，抱怨搜索不好用。
输出：
👤 基本信息：25-30 岁，互联网从业者，一线城市（推测）
📱 使用行为：高频（日均 3+ 次），通勤场景为主，重度使用语音输入
😣 痛点：笔记整理后难以快速检索，语音转文字错误率高
🎯 核心目标：希望用最少操作记录灵感，并能随时找到
💡 一句话总结：追求效率的“速记型”用户，对检索体验敏感

现在请基于以下数据生成画像：
{{behavior_data}}
```

## 模板 5：会议纪要结构化（从录音转写文本）
适用场景：拿到会议录音转写文本（或逐字稿），需要生成包含结论、待办、争议点的结构化纪要。
适用客户端：ChatGPT / Claude / 通义千问
角色：技术会议记录员
上下文：你收到了一次项目评审会的转写文本，需要提炼出关键信息，避免流水账。

约束：
只提取与决策、行动项、风险相关的内容
待办必须包含：责任人（如能识别）、截止时间（如未提及则写“待定”）
不添加个人观点，不评价发言好坏
如果信息冲突，并列标注两种观点
输出格式：按以下标题输出：
📌 会议主题与目标
🧠 关键结论（3-5 条）
✅ 待办事项（表格：事项、责任人、截止时间）
⚠️ 未决争议点（如有）
是否使用进阶技巧：无（但可附加说明 Self-Consistency 适用边界）
变量占位符：
{{transcript}}
{{meeting_name}}

Prompt 正文：

```text
你是一名技术会议记录员。以下是一次「{{meeting_name}}」的录音转写文本，请生成结构化会议纪要。

转写文本：
{{transcript}}

要求：
- 忽略寒暄、重复、离题内容
- 结论必须明确是达成一致的（例如“大家同意…”或“决定…”）
- 待办事项如果责任人未明确，写“待确认”
- 争议点需注明双方观点

输出格式：
📌 会议主题与目标
（一句话概括）

🧠 关键结论
1. ...
2. ...

✅ 待办事项
| 事项 | 责任人 | 截止时间 |
|------|--------|----------|
| ... | ... | ... |

⚠️ 未决争议点
- 争议1：观点A vs 观点B

请直接输出纪要，不要额外解释。
```
