**三次握手**

  

为什么不是两次？为什么不是四次？或者其它更多更少次？

  

TCP传输协议是点到点的，涉及服务端和客户端。

  

握手就是要确认双方的意愿一致且沟通途径有效稳定。即证明双方的发送能力和接收能力没问题。

  

第一次握手🤝：客户端发消息给服务端。

- 当服务端接收到信息时，证明：客户端的发送消息能力没问题。

第二次握手🤝：服务端将回复消息回给客户端。

- 当客户端接收到回复信息时。证明：服务端的接收能力没问题、服务端的发送能力没问题

第三次握手🤝：客户端再次给服务端发送确认消息。

- 当服务端接收到确认消息，证明：发送端的接收能力没问题。

  

至此，双方的发送能力和接收能力都被证明没问题。也才被证明没问题。所以握手得是三次。

怎么识别？

— 逐一标记与压缩。

— 标记：检查系统中所有的对象，分别标记哪些内存空间被使用，哪些是空闲的。

— 压缩：当把某些可被清理的垃圾丢到🗑️后，有些内存空间就free了。由于垃圾的位置随机不定，被清除后，留下凌乱的可用空间。采用压缩，将剩余的对象都推到一起，方便之后新对象的空间分配。

— 工作量大，费时费力还不讨好。很明显啊，对象越多，每一次用时就约长～又因为其实很多对象沦为垃圾的时间很快，触发工作的次数很多。性能就差啊。

**1****、存在重复元素。**

给你一个整数数组 nums 。如果任一值在数组中出现 **至少两次** ，返回 true ；如果数组中每个元素互不相同，返回 false 。

  

**方法一：** **排序**

  

思路：先排序，再判断相邻的是否相同。

  

注： java有自带的排序方法： Arrays.sort(nums); // nums 是数组。

  

**方法二：** **使用** **set**

  

知识基础：

- Set集合具有无序、不重复的特点。
- 实现类为HashSet。
- HashSet的特点：

- 存储唯一元素并允许空值，依据对象的hashcode来确定该元素是否存在。
- 由HashMap支持。
- 不保持插入顺序。
- 非线程安全。
- 性能参数：初始容量、负载因子。分别为16、0.75。

  

思路：新创建一个set集合对象，然后根据set集合元素不重复的特点，遍历数组元素，判断set集合中是否包含同样的元素，返回布尔值。

  

注：利用流Stream实现，会更加简短一些。

- 思路：通过去除数组中的重复元素后数组的长度与之前数组的长度做比较，可得出数组中是否有重复元素。

  

知识储备：

  

- disdinct(): 去除Stream中的重复元素。

  

  

**2****、最大子数组求和**

给你一个整数数组nums，请找出一个具有最大和的连续子数组（子数组最少包含一个元素），返回其最大和。子数组是数组中的一个连续部分。

  

**方法一：分治法**

  

思路：连续子序列的最大和主要由以下三部分子区间里元素的最大和得到：

  

- 子区间：[left, mid]
- 子区间：[mid+1, right]
- 包含子区间[mid, mid+1]的子区间，即nums[mid] 与 nums[mid+1] 一定会被选取。

  

第三部分跨越两个区间的连续子数组的时候，由于nums[mid] 与 nums[mid+1]一定会被选取，可以**从中间向两边扩散，扩散到底选出最大值**。

**3****、两数之和** **Two_Sum**

给定一个整数数组 nums 和一个整数目标值 target，请你在该数组中找出 和为目标值 target  的那 两个 整数，并返回它们的数组下标。你可以假设每种输入只会对应一个答案。但是，数组中同一个元素在答案里不能重复出现。

  

**方法一：暴力枚举法**

  

思路：列出所有可能的组合。

时间复杂度：O(n^2).

  

**方法二：哈希表法**

  

思路：遍历数组，每次计算target-当前索引值的差值。若差值在哈希表中可以找到，则返回当前索引和哈希索引；若找不到，则将当前索引值和当前索引添加在哈希表中。循环。

  

能够快速寻找数组中是否存在目标元素。如果存在，我们需要找出它的索引。

使用哈希表，可以将寻找 target - x 的时间复杂度降低到从 O(n^2) 降低到 O(n)。

限制：数字位数在 1-100 之间；每个节点值在 0-9 之间。

  

方法： 个位开始相加，进位向高位传递。符合手工加法。

object NetworkUrlValidator {

    /**

     * 验证给定的字符串是否为有效的网络地址。

     * 

     * @param input 待验证的字符串。

     * @return 如果是有效的网络地址返回 true，否则返回 false。

     */

    fun isValidNetworkUrl(input: String): Boolean {

        // 定义匹配网络地址的正则表达式

        val urlPattern = Regex(

            "^(?:http|https|ftp|sftp|file)://" + // 匹配协议部分

            "[\\w-]+(\\.[\\w-]+)+" + // 匹配域名部分

            "(?:/[\\w-./?%&=]*)?$" // 匹配路径、查询参数和片段部分

        )

        return urlPattern.matches(input.trim())

    }

}

  

// 对输入字符串进行 trim() 操作，去除首尾空白字符，增强验证的健壮性

  

  

--- 非正则实现，结合Java 的 java.net.URL 类进行验证

object EnhancedNetworkUrlValidator {

    /**

     * 增强验证给定的字符串是否为有效的网络地址。

     * 

     * @param input 待验证的字符串。

     * @return 如果是有效的网络地址返回 true，否则返回 false。

     */

    fun isValidNetworkUrl(input: String): Boolean {

        try {

            val url = URL(input.trim())

            val protocol = url.protocol

            return listOf("http", "https", "ftp", "sftp", "file").contains(protocol)

        } catch (e: MalformedURLException) {

            return false

        }

    }

}

**第一层：什么是** **Flow****？（水管理论）**

在没有 Flow 之前，我们要返回多个值通常用 List。但 List 是一次性的：你必须等水桶装满水，才能整个提走。Flow 就像一根自来水管：1.上游（Producer）：水源地，负责往水管里抽水（发送数据 emit）。2.中游（Operator）：滤水器、加热器，负责对水进行加工（变换数据 map/filter）。3.下游（Consumer）：你家的水龙头，负责接水（接收数据 collect）。特点：数据是流动的，上游发一个，下游接一个，不需要等全部走完。

  

**第二层：冷流（****Cold Flow****）** **vs** **热流（****Hot Flow****）**

这是初学者最容易晕的地方，但非常重要。1. 冷流（Flow）•特性：不听不发。如果没有人在下游接水（collect），上游就不会抽水。•场景：数据库查询、网络请求。•例子：你在代码中没直接用到简单的冷流，但 flow { ... } 块定义的就是冷流。2. 热流（StateFlow / SharedFlow）•特性：不管有没有人听，它都在发。就像广播电台。•在你的代码中：◦StateFlow（如 devices）：它是有状态的。它永远存着最新的一个“值”。新来的听众（UI）一打开收音机，立刻就能听到当前的最新消息。这非常适合用来做 UI 状态刷新。◦SharedFlow（如 _refreshTrigger）：它是事件。它不存状态，只负责把动作发出去。如果你没听的时候发了，那就错过了。这适合用来做“刷新、弹窗、跳转”等一次性指令。

**第四层：****Android** **开发中的生命周期（避坑指南）**

在 Android 里，UI 是会销毁的。如果 UI 销毁了，水管还在流水，就会内存泄漏。1.viewModelScope：你在代码里用得很好。它保证了 ViewModel 销毁时，所有水管自动关掉。2.stateIn：Kotlin.stateIn(viewModelScope, SharingStarted.Eagerly, emptyList())这个操作把一个冷流（水管）变成了热流（蓄水池）。SharingStarted.Eagerly 告诉它：ViewModel 一启动，立刻开始抽水。

  

**第五层：总结****——****给小白的** **Flow** **编程公式**

当你面对一个 Android 需求时，按这个步骤思考：

1.源头：我要监听什么？（传感器、数据库、用户点击）。

2.类型：◦存状态吗？用 StateFlow。◦传指令吗？用 SharedFlow。

3.加工：◦过滤？filter。◦转换？map。◦防抖（防止点太快）？debounce（你已经在刷新逻辑里用到了，非常棒！）。

4.消费：在 UI 层（Compose）使用 collectAsStateWithLifecycle()（这是目前官方最推荐的安全接水方式）。

  

小白金句：Flow 就是一组异步的数据流。上游发，中游变，下游接。

在 Android 里，用 StateFlow 存状态，用 SharedFlow 发动作，准没错！

**理解****substring**

  

每次substring 都会创建一个新的string 对象，在heap 堆里分配一块新的内存。

  

在java 7 之前，不会创建新对象，而是索引到原字符串（新字符串指向原字符串的同一个数组并记录一个偏移量）。这会导致内存泄露。（例如，1GB的字符串，仅被索引一个字符，却不能回收原字符串。）后续版本中，就修改了其机制。
