**ssdp:alive**

- service may declare their presence
- NT must be set to the service's 
- when a ssdp:alive request is received whose USN matches the USN of an entity already in the SSDP client's cache then all information regarding that USN is to be replaced with the information on the ssdp:alive request.
- ssdp:alive requests can be used to update location information and prevent cache entries from expiringl.
- parameters setting formula:

- NT must be set to the service's service type.
- USN header must be contained and it must be set to the SSDP service's USN.
- a Location and/or AL header must be contained.

- when there is no DNS support available on the local network then at least one location should be provided using an ip address of the SSDP service.

- cache-control: max- age or Expires header should be contained.

- if both are present then they are to be processed in the order specified by HTTP/1.1, that is , the cache-control header takes precedence of the expired header.
- if neither the cache-control nor the Expires header is provided the information in the ssdp:alive request must bot be cached by SSDP clients.

**Terminology**

  

- USN:

- 不同服务的统一服务名，提供一种标识出相同类型服务的能力
- unqiue service name
- a URI that is guaranteed to be unique across the entire URI namespace for alll time.
- uniquely identify a particular service.

- MX：

- 设备响应最长等待时间，设备响应在0和这个值之间随机选择响应延迟的时间 — 可以控制点响应平衡网络负载

- ST:

- 设置服务查询的目标
- 类型必须是：

- ssdp:all

- 搜索所有的设备和服务

- upnp:rootdevice

- 搜索网络中的根设备

- uuid:device-UUID

- 查询UUID标识的设备

- urn:schemas-upnp-org:device:device-Type:version

- 查询device-Type字段指定的设备类型，设备类型和版本由UPNP组织定义

- urn:schemas-upnp-org:service:service-Type:version

- 查询device-Type字段指定的服务类型，服务类型和版本由UPNP组织定义

- CACHE-CONTROL

- max-age

- 指定通知消息存活时间，若超过此间隔，控制点可认为设备不存在

- DATE

- 指定响应生成的时间

- EXT

- 向控制点确认 MAN头域已经被设备理解

- LOCATION

- 包含根设备描述的URL地址

- SERVER

- 包含操作系统名，版本、产品名和产品版本信息

- NT

- Notification Type
- 通知类型，值一般是当前设备的类型

- NTS

- Notification Sub-Type

- USN

- 设备的UUID，防止设备的IP或者网络坏境改变后，连接至错误的设备
- 表示不同服务的统一服务名，它提供了一种标识出相同类型服务的能力

**重构名录**

  

**1****、提炼函数（****Extract Function****）**

  

- 何时应该把代码放进独立的函数。

- 将意图与实现分开。
- 被用过不止一次的代码。
- 函数代码过长，超过一屏显示。

  

- 为函数取一个一眼看得出函数在做什么（意图）的名字。

  

- 使用嵌套函数。

  

**2****、内联函数（****Inline Function****）**

  

- 做法：

- 检查函数，确定其不具备多态性。
- 找出所有的调用点。
- 将函数的所有调用点都替换成函数本体。
- 每次替换后，执行测试。
- 删除函数定义。

  

**3****、提炼变量（****Extract Variable****）**

  

- 表达式有可能非常复杂而难以阅读。

- 若变量仅在当前函数有意义，则直接提炼变量。
- 若这个变量会在更宽的上下文也有意义，则可以暴露出来，以函数的形式。
- 若暴露出来后，会需要额外的比较大的工作量，则暂时搁置，稍后以查询取代临时变量。

**5****、改变函数声明（****Change Function Declaration****）**

  

- 如果看到一个函数名字不对，一旦发现更好的名字，就尽快给函数改名。

  

**6****、封装变量（****Encapsulate Variable****）**

  

- 重新组织数据，若作用域范围比较小，则比较容易实现。若作用域范围比较大，例如全局变量，实现起来就比较难。

  

**7****、引入参数对象（****Introduce Parameter Object****）**

  

- 一组数据项总是结伴而行，出没于一个又一个的函数。这样的一组数据就是所谓的数据泥团，可以代之以一个数据结构。

  

8、以对象取代基本类型（Replace Primitive With Object）

  

- 一旦发现对某个数据的操作不仅仅局限于打印时，就为它创建一个新类。
