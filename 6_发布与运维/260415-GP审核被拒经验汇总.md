# Google Play 审核被拒经验汇总

> 常见被拒原因及解决方案 | 2026-04-15 | #发布 #GP #审核 #踩坑

## 🎯 核心原则

**被拒不可怕，可怕的是不知道原因和怎么改。**

每次被拒后，Google 会在 Play Console 的「政策状态」页面给出具体违规说明和修改指引。仔细阅读 + 针对性修改 = 最终通过。

## 📝 被拒类型及解决方案

### 1. 设备和网络滥用

**典型触发场景**：

| 触发行为 | 说明 |
|----------|------|
| 修改系统设置 | 未经用户明确同意修改 WiFi、蓝牙、飞行模式等 |
| 滥用辅助功能 | 使用 AccessibilityService 执行非辅助功能操作 |
| 过度唤醒设备 | 频繁唤醒屏幕或保持系统唤醒锁 |
| 网络欺骗 | 伪造网络状态、劫持流量 |
| **外部更新机制** | 使用 Google Play 以外的渠道更新应用 |
| **动态下载代码** | 从非 Play 来源下载 dex、JAR、.so 等可执行文件 |

**被拒案例**：

| 违规行为 | 具体描述 |
|----------|----------|
| 三方广告源 | 集成了除 AdMob 之外的其他三方广告 SDK |
| 二次重定向 | 应用中存在具有二次重定向跳转的引流链接 |

**解决方案**：

| 步骤 | 操作 |
|------|------|
| 1 | 移除除 AdMob 以外的所有三方广告 SDK |
| 2 | 删除所有具有二次重定向行为的引流链接 |
| 3 | 确保应用更新仅通过 Google Play 分发机制 |
| 4 | 禁止从非官方来源动态加载可执行代码 |

**整改后验证**：

| 验证项 | 检查方法 |
|--------|----------|
| 广告源 | 检查 `build.gradle` 和 `libs/` 目录，确认只有 AdMob |
| 重定向链接 | 遍历所有 WebView 和隐式 Intent，确认无二次跳转 |
| 动态加载 | 检查 `DexClassLoader`、`JNI` 加载路径 |

### 2. 订阅政策违规

**典型触发场景**：

| 触发行为 | 说明 |
|----------|------|
| 订阅引导不清晰 | 未明确说明价格、周期、续费规则 |
| 缺乏退订说明 | 用户不知道如何取消订阅 |
| 强制订阅 | 核心功能强制要求订阅，无免费试用或单次购买选项 |
| 订阅迁移问题 | 跨设备/跨账号订阅状态未同步 |

**解决方案**：

| 步骤 | 操作 |
|------|------|
| 1 | 订阅页面必须清晰展示：价格、计费周期、免费试用时长（如有） |
| 2 | 提供「关闭自动续费」或「取消订阅」的说明链接 |
| 3 | 使用 Google Play Billing Library 标准流程 |
| 4 | 在应用内提供恢复购买的入口 |
| 5 | 遵守「订阅必须在应用内管理，不能引导用户去网页操作」的规定 |

---

### 3. 假冒行为

**典型触发场景**：

| 触发行为 | 说明 |
|----------|------|
| 冒充知名应用 | 应用名、图标、描述与知名应用高度相似 |
| 冒充系统应用 | 伪装成系统设置、清理工具等 |
| 冒充其他开发者 | 包名、开发者名称模仿他人 |

**解决方案**：

| 步骤 | 操作 |
|------|------|
| 1 | 确认应用名称未侵犯他人商标 |
| 2 | 图标不能与其他知名应用混淆 |
| 3 | 描述中明确声明自己的品牌 |
| 4 | 如使用开源代码，遵守 License 并声明来源 |

---

### 4. 商标侵权

**典型触发场景**：

| 触发行为 | 说明 |
|----------|------|
| 使用商标词 | 标题/描述中包含他人注册商标（如 "Facebook"、"WhatsApp 助手"） |
| 暗示官方 | 声称「官方版」、「授权版」但无授权 |
| 图标侵权 | 使用与注册商标相似的图形 |

**解决方案**：

| 步骤 | 操作 |
|------|------|
| 1 | 移除标题/描述中的商标词汇 |
| 2 | 改为「第三方客户端」或「非官方工具」等中性描述 |
| 3 | 修改图标，避免与官方相似 |
| 4 | 如有授权，在 Play Console 提交授权证明 |

---

### 5. 干扰性广告

**典型触发场景**：

| 触发行为 | 说明 |
|----------|------|
| 全屏插屏时机不当 | 应用启动时、用户关闭应用时、游戏进行中弹出 |
| 意外点击 | 广告布局导致用户误触 |
| 模仿系统弹窗 | 广告样式伪装成系统通知或警告 |
| 无法关闭 | 广告没有明显关闭按钮 |

**解决方案**：

| 步骤 | 操作 |
|------|------|
| 1 | 全屏广告只能在**自然间歇点**展示（如关卡结束、结果页） |
| 2 | 禁止在应用启动或退出时弹出插屏 |
| 3 | 广告必须提供清晰可见的关闭按钮 |
| 4 | 广告样式不能模仿系统原生弹窗 |
| 5 | 激励视频需明确告知用户「观看可获得奖励」 |

---

### 6. 权限声明有误

**典型触发场景**：

| 触发行为 | 说明 |
|----------|------|
| 未声明权限用途 | 使用敏感权限但未在 Play Console 填写说明 |
| 声明与实际不符 | 填写的用途与代码实际行为不一致 |
| 过度声明 | 申请了非核心功能需要的权限 |

**解决方案**：

| 步骤 | 操作 |
|------|------|
| 1 | 在 Play Console → 政策 → 应用内容 → 敏感权限 中声明 |
| 2 | 声明内容需与代码行为一致（如声明「用于拍照」就不能用于上传联系人） |
| 3 | 仅在用户触发对应功能时请求权限，避免启动时一次性请求所有权限 |
| 4 | 视频演示：上传应用操作录屏，证明权限用途与声明一致 |

**常见需要声明的权限**：

| 权限 | 常见合理用途 |
|------|--------------|
| 位置 | 地图导航、天气、本地推荐 |
| 相机 | 拍照、扫码、视频通话 |
| 麦克风 | 录音、语音输入 |
| 联系人 | 通讯录备份、邀请好友 |
| 存储 | 保存文件、读取图片 |

## 📝 被拒后的标准处理流程

| 步骤 | 操作 |
|------|------|
| 1 | 查看 Play Console → 政策状态 → 具体违规说明 |
| 2 | 分析触发原因，确定是代码问题还是商店信息问题 |
| 3 | 修改代码 / 更新商店信息 |
| 4 | 在 Play Console 提交新版本 |
| 5 | 如有异议，可通过「申诉」按钮提交解释 |

## ⚠️ 注意事项

1. 多次严重违规可能导致**开发者账号封禁**
2. 被拒后不要只改商店信息，检查代码中是否真的有违规行为
3. 权限声明视频需清晰展示：启动 → 触发权限 → 使用功能的完整流程
4. 订阅问题修改后，需确保老用户也能正常使用
5. 广告问题修改后，建议录屏证明广告展示时机已调整
